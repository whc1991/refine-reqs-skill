# refine-reqs

用于审阅、澄清和细化软件需求，把没说清楚的事情想清楚，并最终形成经确认的需求规格说明（SPEC）和功能点 PR 列表。

## 名称说明

```yaml
name: refine-reqs
```

- `refine`：明确表示澄清、细化和收敛；
- `reqs`：软件工程中通常是 `requirements` 的复数缩写；
- `refine-reqs`：采用“动词 + 对象”的命名方式，表达“细化需求”，含义直接；
- 使用复数 `reqs`，比单独的 `req` 更容易理解为需求集合，降低与 `request`、HTTP 请求对象等概念混淆的可能；
- 名称较短、容易拼写，适合频繁显式调用。

## 最简使用

调用 `$refine-reqs` 已经代表执行完整的需求分析流程。用户只需提供分析对象，不需要重复说明“分析需求”“澄清需求”“拆分 PR”或“不要开发”。

### 单个文档

```text
$refine-reqs docs/工单转派需求.md
```

### 多个文档

```text
$refine-reqs docs/PRD-v2.docx docs/接口说明.pdf docs/会议纪要.md
```

未指定主次时，Skill 会综合审阅所有材料，不默认第一份是主文档，也不会先要求用户指定主文档。只有材料存在无法自行消解且会影响范围或验收的实质冲突时，才会询问以哪项为准。

### 对比版本

```text
$refine-reqs 对比 docs/需求-v1.md docs/需求-v2.md
```

这里保留“对比”，因为它改变了默认分析重点，要求优先识别版本差异、冲突和缺失内容。

### 整个目录

```text
$refine-reqs docs/requirements/
```

### 只有一个需求想法

```text
$refine-reqs 工单需要支持转派
```

没有文档时，Skill 会从这句话建立初始需求基线，再逐轮澄清。

## 何时补充指令

通常无需在调用时添加额外提示。只有需要改变默认行为时才补充，例如：

```text
$refine-reqs docs/requirements/ 只梳理一期范围
```

```text
$refine-reqs docs/PRD.md 重点核对权限和异常流程
```

```text
$refine-reqs docs/requirements/ 排除 docs/requirements/archive/
```

## 默认流程

```text
审阅材料
-> 建立需求基线
-> 识别遗漏、歧义和冲突
-> 每轮只确认一个关键问题
-> 输出 SPEC 并等待确认
-> 拆分功能点 PR 列表
-> 结束
```

每个澄清问题都会提供推荐答案或选项，并说明推荐依据和主要取舍。Skill 会优先从文档和项目上下文中寻找答案，只询问无法可靠推断且需要用户决策的事项。

## 支持的输入

- PRD、需求说明、用户故事和验收标准；
- Markdown、TXT、PDF、DOCX 和表格；
- 原型、流程图、截图和会议纪要；
- API 文档、数据字典和第三方系统说明；
- 项目代码、配置、测试和历史文档；
- 对话中的需求想法和补充决定。

## 输出

需求收敛后，Skill 会先输出带需求编号和来源追踪的 SPEC。只有用户明确确认 SPEC 后，才会继续输出可独立评审、验收和排序的功能点 PR 列表。

这里的 PR 表示拟定的功能交付单元，不是真实创建的 Pull Request。

## 边界

本 Skill 不设计技术方案，不输出代码、SQL 或架构图，不修改项目文件，也不进入开发实施。编程语言和框架不会被预设，只作为已有或待确认的技术约束。
