# refine-reqs

用于审阅、澄清、固化和冻结软件需求，把没说清楚的事情想清楚，并形成带决策、需求、约束、验收和功能点完整映射的 SPEC 与 PR 列表。

## 名称说明

```yaml
name: refine-reqs
```

- `refine`：明确表示澄清、细化和收敛；
- `reqs`：软件工程中通常是 `requirements` 的复数缩写；
- `refine-reqs`：采用“动词 + 对象”的命名方式，表达“细化需求”，含义直接；
- 使用复数 `reqs`，比单独的 `req` 更容易理解为需求集合，降低与 `request`、HTTP 请求对象等概念混淆的可能；
- 名称较短、容易拼写，适合频繁显式调用。

## 安装

将下面示例中的 `<github-user>` 替换为仓库所有者的 GitHub 用户名或组织名。

### 使用 Codex 安装（推荐）

在 Codex 中输入：

```text
$skill-installer 安装 https://github.com/<github-user>/refine-reqs
```

安装完成后新开一个会话，即可调用 `$refine-reqs`。

### 手动安装

将仓库克隆到个人 Codex Skills 目录：

```bash
git clone https://github.com/<github-user>/refine-reqs.git ~/.codex/skills/refine-reqs
```

安装后新开一个 Codex 会话。如果 `~/.codex/skills/refine-reqs` 已存在，请先自行备份或改名；不要直接覆盖已有版本。

### 验证目录

安装后的目录至少应包含：

```text
~/.codex/skills/refine-reqs/
├── SKILL.md
├── README.md
└── agents/
    └── openai.yaml
```

## 最简使用

调用 `$refine-reqs` 已经代表执行完整的需求分析流程。用户只需提供分析对象，不需要重复说明“分析需求”“澄清需求”“拆分 PR”或“不要开发”。

### 单个文档

```text
$refine-reqs docs/工单转派需求.md
```

### 多个文档

```text
$refine-reqs docs/PRD-v2.docx docs/接口说明.pdf docs/会议纪要.md
```

未指定主次时，Skill 会综合审阅所有材料，不默认第一份是主文档，也不会先要求用户指定主文档。只有材料存在无法自行消解且会影响范围或验收的实质冲突时，才会询问以哪项为准。

### 对比版本

```text
$refine-reqs 对比 docs/需求-v1.md docs/需求-v2.md
```

这里保留“对比”，因为它改变了默认分析重点，要求优先识别版本差异、冲突和缺失内容。

### 整个目录

```text
$refine-reqs docs/requirements/
```

### 只有一个需求想法

```text
$refine-reqs 工单需要支持转派
```

没有文档时，Skill 会从这句话建立初始需求基线，再逐轮澄清。

## 何时补充指令

通常无需在调用时添加额外提示。只有需要改变默认行为时才补充，例如：

```text
$refine-reqs docs/requirements/ 只梳理一期范围
```

```text
$refine-reqs docs/PRD.md 重点核对权限和异常流程
```

```text
$refine-reqs docs/requirements/ 排除 docs/requirements/archive/
```

## 默认流程

```text
审阅材料
-> 建立需求基线
-> 识别遗漏、歧义和冲突
-> 每轮只确认一个关键问题
-> 每次拍板后立即登记并回显 DC-xxx
-> 输出《初步需求冻结书》
-> 等待用户回复“确认并冻结”
-> 输出冻结 SPEC
-> 拆分并校验功能点 PR 列表
-> 询问是否将冻结 SPEC、PR 列表或两者落地到文件
-> 用户确认后写入，或直接结束
```

每个澄清问题都会提供推荐答案或选项，并说明推荐依据和主要取舍。Skill 会优先从文档和项目上下文中寻找答案，只询问无法可靠推断且需要用户决策的事项。

## 即时固化与冻结

用户每次作出明确决定后，Skill 会先回显一个对话内的临时标记块，再继续提问：

```text
[CONFIRMED_REQUIREMENTS.md]
【固化确认】
- DC-001｜主题：工单超时｜用户原答：“改成30分钟”｜固化结论：工单超时关闭阈值为30分钟｜拟落点：C-001｜状态：已确认
[/CONFIRMED_REQUIREMENTS.md]
```

这个标记块不是实际项目文件。需求基本收敛后，Skill 会输出《初步需求冻结书》，并且只有收到完全一致的 `确认并冻结` 指令后才会形成冻结基线。每次请求该口令时都必须在同一回复完整展示当前冻结书和映射，不能只索要口令。冻结后新增或修改业务需求时，需要解冻、修订并重新冻结，不能静默追加。

## 编号体系

| 编号 | 含义 |
|---|---|
| `DC-xxx` | 用户在澄清中明确确认的决策 |
| `F-xxx` | 系统必须提供的功能需求 |
| `C-xxx` | 开发和交付不得违反的硬性约束 |
| `AC-xxx` | 可执行、可判定的验收标准 |
| `PR-xx` | 可独立开发和验收的功能单元 |

已有材料中的 `REQ-xxx` 可以保留，新产物使用上述编号并记录兼容映射。

## 三段追踪映射

### 决策到需求

| 确认决策 | 用户结论 | SPEC 落点 |
|---|---|---|
| `DC-001` | 超时关闭阈值为30分钟 | `C-001` |
| `DC-002` | 支持待处理工单撤回 | `F-002`、`C-002` |
| `DC-003` | 暂不接入邮件通知 | `Non-goal-01` |

每个有效 `DC-xxx` 必须映射到功能需求、硬性约束、验收标准、Non-goal、范围或其他明确章节。不能静默丢弃，也不能强制把所有决定都误分类为约束。

### 需求到开发单元

| 功能开发点 | 冻结基线 | 对应功能 | 适用约束 | 确认决策 | 验收标准 |
|---|---|---|---|---|---|
| `PR-01` 超时处理 | `SPEC v1.0` | `F-001` | `C-001` | `DC-001` | `AC-001` |
| `PR-02` 工单撤回 | `SPEC v1.0` | `F-002` | `C-002` | `DC-002` | `AC-002` |

每个功能、约束和确认决策都必须被适用 PR 显式覆盖。全局约束必须挂载到所有适用 PR，而不是只出现在其中一个 PR。

Non-goal、沿用现状或其他不产生开发工作的决定，必须在映射中明确写为“无开发单元”并说明原因，不能静默消失，也不能为了通过检查制造虚假 PR。

### 开发单元到实现证据

该映射由 `dev-flow` 在实现完成后生成：

| 需求或约束 | 开发单元 | 实现证据 | 验证证据 | 状态 |
|---|---|---|---|---|
| `F-001` | `PR-01` | `app/service.py: process_timeout` | `tests/test_timeout.py: test_timeout_close` | 通过 |
| `C-001` | `PR-01` | `app/config.py: timeout_minutes` | `tests/test_timeout.py: test_closes_at_30_minutes` | 通过 |

因此完整链路为：

```text
用户问题与回答
-> DC-xxx
-> F-xxx / C-xxx / AC-xxx / Non-goal
-> PR-xx
-> 文件:符号 / 配置 / 数据库对象
-> 测试或其他验证证据
```

## 支持的输入

- PRD、需求说明、用户故事和验收标准；
- Markdown、TXT、PDF、DOCX 和表格；
- 原型、流程图、截图和会议纪要；
- API 文档、数据字典和第三方系统说明；
- 项目代码、配置、测试和历史文档；
- 对话中的需求想法和补充决定。

## 输出

需求收敛后，Skill 会先输出冻结书。用户回复 `确认并冻结` 后，冻结书成为最终 SPEC 的唯一内容基线，再生成可独立评审、验收和排序的功能点 PR 列表。分析完成后，Skill 会询问是否落地冻结 SPEC、PR 列表或两者；只有用户明确同意后才会写入。

跨会话交给 `dev-flow` 时，必须同时落地冻结 SPEC 和 PR 列表，或将两者合并到一个文件。只保存其中一项不构成完整开发交接。

这里的 PR 表示拟定的功能交付单元，不是真实创建的 Pull Request。

## 边界

本 Skill 不设计技术方案，不输出代码、SQL 或架构图，也不进入开发实施。`确认并冻结` 只确认需求基线，不授权编写代码。除用户明确同意落地的需求分析产物外，不修改项目文件。编程语言和框架不会被预设，只作为已有或待确认的技术约束。
